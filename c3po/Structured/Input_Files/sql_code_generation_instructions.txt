CRITICAL ERROR PREVENTION:
- Never use INTERVAL keyword anywhere
- Never use three-parameter date functions
- Month operations: ADD_MONTHS(date_col, n) only
- Day operations: DATE_ADD(date_col, n) or DATE_SUB(date_col, n) only
- Quote names with hyphens using backticks: catalog-name.schema.table
- COLUMN REFERENCE FIX: When using nested queries/subqueries, always include ALL relevant columns required for analysis in the parent SELECT statement, especially those that will be referenced or filtered in subsequent child queries, to avoid "column not found" errors

-  Follow Logical Steps:
Use a structured approach: Once the relevant columns are identified, Apply filters → Group data → Perform aggregations → Sort or rank as needed.

1. FILTERING & PATTERN MATCHING
- Use ILIKE for all case-insensitive comparisons  
  WHERE customer_name ILIKE '%smith%'

2. DATABRICKS SQL - DATES & TIMES REQUIREMENTS
    Date Grouping & Formatting:
    - Grouping by month: DATE_TRUNC(date_col, 'month')
    - Custom formatting: DATE_FORMAT(date_col, 'yyyy-MM')
    - Example:
    SELECT 
        MAX(week_end_date) AS latest_date,
        DATE_FORMAT(MAX(week_end_date), 'yyyy-MM-dd') AS latest_date_str
    FROM catalog.schema.table;

    Date Arithmetic - MANDATORY RULES:
    - Month operations: Use ADD_MONTHS(date_col, n) with integer months
    - Format: ADD_MONTHS(date, -1) where -1 is direct integer, NOT INTERVAL
    - First day of year: DATE_TRUNC('YEAR', date_col) (cast to STRING if needed)
    - Prefer calendar-aware functions: ADD_MONTHS, DATE_DIFF over hard-coded day counts

    String-to-Date Conversion - CRITICAL:
    - ALWAYS use DATE() to convert string dates before arithmetic
    - Wrong: Month_Year - INTERVAL 1 MONTH
    - Correct: ADD_MONTHS(DATE(Month_Year), -1)

    Required Functions:
    - Month arithmetic: ADD_MONTHS(date, -1)
    - String to date: DATE(column) or TO_DATE(column, 'format')
    - NEVER use: DATE_SUB with INTERVAL on strings

    Table References - MANDATORY:
    - Always use explicit aliases
    - Wrong: SELECT Month_Year FROM table
    - Correct: SELECT h.Month_Year FROM table h

    Data Types - REQUIRED:
    - Specify lengths explicitly: VARCHAR(50), INT, DATE

    REJECT any SQL that violates these rules.

3. SUBQUERIES, CTEs & ALIASING

- Wrap scalar subqueries in parentheses:  WHERE amt > (SELECT …)
- Keep SQL queries simple and avoid wrapping them in unnecessary SELECT * FROM (...) constructs
- Alias every CTE / derived table clearly: … AS sales_by_week
- Debug "no-rows" issues by running each CTE individually.
- Apply ORDER BY only at the outermost level when final ordering is required.
- Use clear aliases for columns and tables.

4. DATA TYPES & NULL HANDLING

- Resolve mismatches with  CAST(expr AS target_type)  or  TRY_CAST(expr AS type)  (returns NULL on failure).
- Aggregate NULLs safely:
    SELECT COALESCE(SUM(sales_amt), 0) AS total_sales
    FROM   orders;
  (Do not wrap the input column with COALESCE inside SUM.)
- Use IS NULL / IS NOT NULL; never = NULL.
- Ensure date columns are DATE or TIMESTAMP before sorting/joining.
- No Integer Casting: Do not cast patient count values to integer. Keep them in their numeric (possibly decimal) form. Casting could lose important fractional data that might result from projections or averaging.
- Always cast to int for patient age before applying any filters.

5. AGGREGATIONS & LOGIC

- List column names in GROUP BY; avoid GROUP BY 1,2.
- Simple two-way logic:     IF(condition, true_val, false_val)
- Multi-branch / searched:  CASE WHEN … THEN … ELSE … END
- Check exact column names from schema
- For window functions with aggregations:
  a. All non-aggregated columns in SELECT must appear in GROUP BY
  b. When using window functions like SUM() OVER(), include all non-aggregated SELECT columns in the PARTITION BY clause
  c. For time-series analysis (R3M/R6M/R12M):
     - GROUP BY all time-related columns (Month_Year) first
     - Include time columns in PARTITION BY for correct rolling calculations
- STRICTLY Use window functions with OVER(PARTITION BY column) instead of GROUP BY when getting aggregate values, and add DISTINCT to eliminate duplicates.

6. JOINS, RANGES & SAFETY CHECKS

- Qualify or alias every join column to avoid ambiguity.
- Use BETWEEN only for inclusive ranges.
- Verify wide date filters and watch for future-dated records.
- Let the application handle empty result sets; add dummy fallback rows only when strictly required (and document why).

7. STYLE & READABILITY

- Quote identifiers only when needed (spaces, digits, reserved words).
- Keep keyword case consistent (SELECT, FROM, WHERE, …).
- Create simplified diagnostic versions of complex queries for troubleshooting.

8. CATALOG/DATABASE/TABLE NAMING

10. Date Calculation Rules 
    - No hardcoded dates (2025-02-01, etc.)
    - All month arithmetic uses ADD_MONTHS()

    R13W (Recent 13 Weeks):   
        - Start date: 12 weeks before the latest `Month_Year`.   
        - End date: The latest `Month_Year`.   
        - Total weeks: 13 (includes start and end weeks).   
    P13W (Prior 13 Weeks):   
        - Start date: 25 weeks before the latest `Month_Year`.   
        - End date: 13 weeks before the latest `Month_Year`.   
        - Total weeks: 13.  
    R12M (Recent 12 Months):   
        - Start date: 51 weeks before the latest `Month_Year`.   
        - End date: The latest `Month_Year`.   
        - Total weeks: 52 (approximately 12 months).   
    P12M (Prior 12 Months):   
        - Start date: 103 weeks before the latest `Month_Year`.   
        - End date: 52 weeks before the latest `Month_Year`.   
        - Total weeks: 52. 
    QTD (Quarter-to-Date): 
        - Start date: First day of the current quarter (e.g., Q1TD starts on  Jan 1, Q2TD on Apr 1, Q3TD on Jul 1, and Q4TD on Oct 1). 
        - End date: Latest Month_Year available in the data.  
    YTD (Year-to-Date): 
        - Start date: January 1 of the current year. 
        - End date: Latest Month_Year. 

    Date Calculation Guidelines
        - Do NOT use the current date for any calculations, whenever applicable.

11. SQL Best Practices to Avoid Common SQL Warehouse Errors :
    a. Data Type Specifications: Always specify length parameters for VARCHAR (e.g., VARCHAR(10)), cast data types explicitly when performing operations between different types, and specify precision for DECIMAL types when required
    b. Column and Table References: Always use explicit table aliases for column references, ensure all referenced columns exist in their respective tables/CTEs, and use aliases to distinguish columns with same names across tables
    c. Date and Time Operations: Use warehouse-specific date functions (DATE_SUB, DATE_ADD) instead of generic functions like ADD_MONTHS, verify INTERVAL syntax matches your SQL dialect, and ensure date functions receive correct parameter types

FORBIDDEN DATE SYNTAX:
- Never use INTERVAL keyword anywhere
- Never use three-parameter date functions
- Never use DATE_ADD/DATE_SUB with INTERVAL syntax

REQUIRED DATE SYNTAX:
- Month operations: ADD_MONTHS(date_col, n)
- Day operations: DATE_ADD(date_col, n) or DATE_SUB(date_col, n)
- Use only 2-parameter format for date functions

COMMON CORRECTIONS:
- DATE_ADD(col, INTERVAL -2 MONTH) → ADD_MONTHS(col, -2)
- DATE_SUB(col, INTERVAL 1 DAY) → DATE_SUB(col, 1)
- DATE_ADD(col, INTERVAL -1 DAY) → DATE_ADD(col, -1)

By following these rules—consistent ILIKE filtering, precise date handling, explicit aliasing, safe type management, and disciplined NULL treatment—you'll write Databricks-friendly SQL that is portable, readable, and robust.
