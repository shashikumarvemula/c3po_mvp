SQL USAGE GUIDELINES
====================

-  Follow Logical Steps:
Use a structured approach: Once the relevant columns are identified, Apply filters → Group data → Perform aggregations → Sort or rank as needed.

NOTE :  NEVER USE THE CURRENT_DATE() FUNCTION ALWAYS USE THE LATEST DATE FROM THE DATA SET.

1. FILTERING & PATTERN MATCHING

- Use ILIKE for all comparisons  
  WHERE customer_name ILIKE '%smith%'

2. DATES & TIMES

- Grouping by month:           DATE_TRUNC(date_col, 'month')
- Formatting custom strings:   DATE_FORMAT(date_col, 'yyyy-MM')
- Example:
    SELECT
      MAX(week_end_date)                            AS latest_date,
      DATE_FORMAT(MAX(week_end_date), 'yyyy-MM-dd') AS latest_date_str
    FROM catalog.schema.table;

- Use DATE_SUB(date_col, n) / DATE_ADD(date_col, n) with integer days (avoid INTERVAL).
- First day of year:          DATE_TRUNC('YEAR', date_col)
  (cast to STRING afterwards if needed).
- Prefer calendar-aware shifts (ADD_MONTHS, DATE_DIFF) over hard-coded day counts.
- For DATE_SUB, use format DATE_SUB(date, n) where n is a direct integer value, not an INTERVAL

3. SUBQUERIES, CTEs & ALIASING

- Wrap scalar subqueries in parentheses:  WHERE amt > (SELECT …)
- Keep SQL queries simple and avoid wrapping them in unnecessary SELECT * FROM (...) constructs
- Alias every CTE / derived table clearly: … AS sales_by_week
- Debug “no-rows” issues by running each CTE individually.
- Apply ORDER BY only at the outermost level when final ordering is required.
- Use clear aliases for columns and tables.

4. DATE CALCULATION RULES

  CRITICAL: Always use the latest available date from the specific table being queried (MAX date from fill_dt for RX, MAX date from svc_dt for MX). NEVER use current date for calculations.

  a. DATE EXCLUSION LOGIC FOR "PAST X" QUERIES:

    **"Past 12 months" or "Past year":**
    - Exclude last 2 incomplete months
    - Start date: DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -13))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -2))
    - This gives exactly 12 complete months

    **"Past X months" (where X is any number):**
    - Exclude last 2 incomplete months
    - Start date: DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1)))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -2))
    
    **"Past X quarters":**
    - Exclude last incomplete quarter
    - Start date: DATE_TRUNC('QUARTER', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1)*3))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -3))

  b. NO DATE RESTRICTIONS FOR:
    - "Monthly trends" (without "past X months")
    - "Quarterly trends" (without "past X quarters")
    - "Since launch"
    - General aggregations like "total claims", "all claims"
    - Questions not specifying "past X" timeframes

  c. MANDATORY DATE CALCULATION PATTERN:

    **For RX queries:**
    ```sql
    WHERE rx.fill_dt BETWEEN 
      DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(fill_dt) FROM table)), -(X+1)))
    AND 
      LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(fill_dt) FROM table)), -2))
    ```

    **For MX queries:**
    ```sql
    WHERE mx.svc_dt BETWEEN 
      DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(svc_dt) FROM table)), -(X+1)))
    AND 
      LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(svc_dt) FROM table)), -2))
    ```

    **For COMBINED (RX + MX) queries:**
    - Calculate date ranges SEPARATELY for each table using their own MAX dates
    - DO NOT use GREATEST() to combine max dates
    - DO NOT use CTEs with date_range calculations
    - Each subquery should have its own WHERE clause with independent date calculations

  d. FORBIDDEN PATTERNS (DO NOT USE):
    - ❌ CTEs with GREATEST(MAX(fill_dt), MAX(svc_dt))
    - ❌ DATE_SUB() function for "past X months" queries
    - ❌ Shared date_range CTEs for combined queries
    - ❌ INTERVAL calculations for "past X months"
    - ❌ Using current_date or CURRENT_DATE()  
  e. R3M average (Rolling 3-Month):
      -If "R3M average for every month" (or similar) is requested, calculate a rolling 3-month average.
      -Filter data as needed (e.g., for 2021, include data from November 2020).
      -Apply a 3-month rolling window to both sums (min_periods=3).
      -Divide the regimen-specific rolling sum by the total rolling sum to get the R3M average for each month.
        
  f. average R3M (in trend questions):
    - Filter data using: WHERE <respective_date_column> BETWEEN add_months('<start_date>', -2) AND '<end_date>' to ensure enough data for the first R3M value.
    - Use a subquery with AVG() over the last 3 months instead of JOIN to prevent over-counting.

  g. R13W (Recent 13 Weeks):  ​
    - Start date: 12 weeks before the latest `<respective_date_column>`.  ​
    - End date: The latest `<respective_date_column>`.  ​
    - Total weeks: 13 (includes start and end weeks).  ​
    P13W (Prior 13 Weeks):  ​
    - Start date: 25 weeks before the latest `<respective_date_column>`.  ​
    - End date: 13 weeks before the latest `<respective_date_column>`.  ​
    - Total weeks: 13. ​
    R12M (Recent 12 Months):  ​
    - Start date: 51 weeks before the latest `<respective_date_column>`.  ​
    - End date: The latest `<respective_date_column>`.  ​
    - Total weeks: 52 (approximately 12 months).  ​
    P12M (Prior 12 Months):  ​
    - Start date: 103 weeks before the latest `<respective_date_column>`.  ​
    - End date: 52 weeks before the latest `<respective_date_column>`.  ​
    - Total weeks: 52.​
    QTD (Quarter-to-Date):​
    - Start date: First day of the current quarter (e.g., Q1TD starts on  Jan 1, Q2TD on Apr 1, Q3TD on Jul 1, and Q4TD on Oct 1).​
    - End date: Latest <respective_date_column>. ​
    YTD (Year-to-Date):​
    - Start date: January 1 of the current year.​
    - End date: Latest <respective_date_column>.​

    Date Calculation Guidelines
    - Do NOT use the current date for any calculations.
    whenever applicable.
  i. ALWAYS USE DATE_TRUNC() METHOD FOR MONTHLY TREND CALCULATIONS.
  j. Complete Time Period Rule: Always exclude incomplete time periods - for "past X months" use complete months only, ending at last day of previous complete month, not current incomplete month.
    Calculation Method: Use LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -1)) for end date and DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1))) for start date to ensure complete month boundaries.

5. AGGREGATIONS & LOGIC

- Add the EXTRACT(YEAR FROM TO_DATE(CAST(Year_Month AS STRING), 'yyyyMM')) expression in the GROUP BY clause: This ensures that the year column used for aggregation is properly included in the GROUP BY statement.
- List column names in GROUP BY; avoid GROUP BY 1,2.
- Simple two-way logic:     IF(condition, true_val, false_val)
- Multi-branch / searched:  CASE WHEN … THEN … ELSE … END
- Check exact column names from schema
- For window functions with aggregations:
  a. All non-aggregated columns in SELECT must appear in GROUP BY
  b. When using window functions like SUM() OVER(), include all non-aggregated SELECT columns in the PARTITION BY clause
  c. For time-series analysis (R3M/R6M/R12M):
     - GROUP BY all time-related columns (Year_Month) first
     - Include time columns in PARTITION BY for correct rolling calculations

6. JOINS, RANGES & SAFETY CHECKS

- Qualify or alias every join column to avoid ambiguity.
- Use BETWEEN only for inclusive ranges.
- Verify wide date filters and watch for future-dated records.
- Let the application handle empty result sets; add dummy fallback rows only when strictly required (and document why).

7. STYLE & READABILITY

- Quote identifiers only when needed (spaces, digits, reserved words).
- Keep keyword case consistent (SELECT, FROM, WHERE, …).
- Create simplified diagnostic versions of complex queries for troubleshooting.

8. CATALOG/DATABASE/TABLE NAMING

- Always quote catalog, schema, and table names with backticks (`) when they contain:
  * Hyphens (-)
  * Spaces
  * Special characters
  * Numbers at the start
- Example format: 
  `catalog-name`.`schema_name`.`table-name`

9. QUERY VALIDATION CHECKLIST

Before executing any SQL query, STRICTLY verify:
✓ Always check the unique values in a column before applying filters to ensure you're filtering for the correct values. 
This avoids errors like filtering with incorrect or incomplete conditions.
✓ Each CTE has proper closing parentheses
✓ No duplicate statements or extra SELECT wrappers
✓ Proper quoting of catalog/schema/table names with backticks (`)
✓ All aggregations have corresponding GROUP BY clauses
✓ Date formats follow the specified standards
✓ Window functions have proper PARTITION BY and ORDER BY clauses

By following these rules—consistent ILIKE filtering, precise date handling, explicit aliasing, safe type management, and disciplined NULL treatment—you’ll write Databricks-friendly SQL that is portable, readable, and robust.

  - Rows with NULL mappings to be included incorrectly

  RULE 2: LEFT JOIN STRUCTURE FOR DRUG FILTERING
  -----------------------------------------------
  ALWAYS use this pattern for RX drug filtering:
      FROM rx_table rx
      LEFT JOIN ndc_mapping ndc ON rx.ndc11 = ndc.product_code
