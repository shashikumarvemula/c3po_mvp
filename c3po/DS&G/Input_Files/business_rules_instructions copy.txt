3. CRITICAL RX+MX COMBINATION RULES (MANDATORY):

  - WHEN USER ASKS about combined RX+MX data (e.g., "total claims", "RX+MX claims", "combined claims") for NON-DRUG-SPECIFIC queries, 
    ALWAYS calculate RX and MX SEPARATELY first, then combine by matching dimension (month/quarter/drug):
    - Calculate RX claims with RX's own latest pt_cycle_id
    - Calculate MX claims with MX's own latest pt_cycle_id  
    - Use FULL OUTER JOIN to combine by claim_month/dimension
    - NEVER use COALESCE on pt_cycle_ids or join RX and MX cycle IDs directly

  - For BATCH COMPARISONS with RX+MX combined:
    - Preprocessed tables now contain latest 3 batches of data
    - Identify latest 3 pt_cycle_ids from RX table independently
    - Identify latest 3 pt_cycle_ids from MX table independently
    - Batch comparisons can be done between any of the 3 latest batches as specified by user
    - Default batch comparison (if not specified): Compare 2nd latest vs latest batch
      - Batch 1 = RX's 2nd latest cycle monthly + MX's 2nd latest cycle monthly (combined by month)
      - Batch 2 = RX's latest cycle monthly + MX's latest cycle monthly (combined by month)
    - Then compare Batch 1 vs Batch 2

  - RX and MX have DIFFERENT cycle IDs and load schedules - treat them as independent data sources 
    that must be aggregated separately before combining results on common dimensions like month or drug.

  - This separation rule applies to ALL queries mentioning "RX+MX", "total claims", "combined", 
    or any analysis requiring both data sources together for NON-DRUG-SPECIFIC queries.

      ```sql
      WHERE rx.fill_dt <= LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(fill_dt) FROM rx_table)), -2))
      ```
    
    - **For MX queries:**
      ```sql
      WHERE mx.svc_dt <= LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(svc_dt) FROM mx_table)), -2))
      ```

  c. WHEN TO OVERRIDE THE TWO-MONTH LAG (ONLY THESE CASES):
    - User explicitly requests "latest data"
    - User explicitly requests "most recent data"
    - User explicitly requests "current month" or "this month"
    - User explicitly states "including incomplete months"
    - User explicitly asks for data "as of today" or "up to now"

  d. EXAMPLES OF TWO-MONTH LAG APPLICATION:

    **Example 1: Simple Count Query**
    User asks: "How many claims for TRODELVY?"
    ŌåÆ Apply two-month lag even though no time period mentioned
    ```sql
    WHERE mx.svc_dt <= LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH',

    ```

    **Example 2: Monthly Trends Without Time Specification**


8. DATE CALCULATION RULES

  CRITICAL: Always use the latest available date from the specific table being queried (MAX date from fill_dt for RX, MAX date from svc_dt for MX). NEVER use current date for calculations.

  a. DATE EXCLUSION LOGIC FOR "PAST X" QUERIES:

    **"Past 12 months" or "Past year":**
    - Exclude last 2 incomplete months
    - Start date: DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -13))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -2))
    - This gives exactly 12 complete months

    **"Past X months" (where X is any number):**
    - Exclude last 2 incomplete months
    - Start date: DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1)))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -2))
    
    **"Past X quarters":**
    - Exclude last incomplete quarter
    - Start date: DATE_TRUNC('QUARTER', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1)*3))
    - End date: LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -3))

  b. DATE RESTRICTIONS WITH TWO-MONTH LAG:
    - "Monthly trends" ŌåÆ Apply two-month lag
    - "Quarterly trends" ŌåÆ Apply two-month lag (or last complete quarter)
    - "Since launch" ŌåÆ Apply two-month lag at the end date
    - General aggregations ŌåÆ Apply two-month lag
    - ONLY exclude two-month lag when user explicitly requests "latest" or "most recent" data

  c. MANDATORY DATE CALCULATION PATTERN:

    **For RX queries:**
    ```sql
    WHERE rx.fill_dt BETWEEN 
      DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(fill_dt) FROM table)), -(X+1)))
    AND 
      LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(fill_dt) FROM table)), -2))
    ```

    **For MX queries:**
    ```sql
    WHERE mx.svc_dt BETWEEN 
      DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(svc_dt) FROM table)), -(X+1)))
    AND 
      LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
        (SELECT MAX(svc_dt) FROM table)), -2))
    ```

    **For COMBINED (RX + MX) queries:**
    - Calculate date ranges SEPARATELY for each table using their own MAX dates
    - DO NOT use GREATEST() to combine max dates
    - DO NOT use CTEs with date_range calculations
    - Each subquery should have its own WHERE clause with independent date calculations

  d. FORBIDDEN PATTERNS (DO NOT USE):
    - ŌØī CTEs with GREATEST(MAX(fill_dt), MAX(svc_dt))
    - ŌØī DATE_SUB() function for "past X months" queries
    - ŌØī Shared date_range CTEs for combined queries
    - ŌØī INTERVAL calculations for "past X months"
    - ŌØī Using current_date or CURRENT_DATE()

  e. R13W (Recent 13 Weeks):  ŌĆŗ
    - Start date: 12 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - End date: The latest `<respective_date_column>`.  ŌĆŗ
    - Total weeks: 13 (includes start and end weeks).  ŌĆŗ
    P13W (Prior 13 Weeks):  ŌĆŗ
    - Start date: 25 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - End date: 13 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - Total weeks: 13. ŌĆŗ
    R12M (Recent 12 Months):  ŌĆŗ
    - Start date: 51 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - End date: The latest `<respective_date_column>`.  ŌĆŗ
    - Total weeks: 52 (approximately 12 months).  ŌĆŗ
    P12M (Prior 12 Months):  ŌĆŗ
    - Start date: 103 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - End date: 52 weeks before the latest `<respective_date_column>`.  ŌĆŗ
    - Total weeks: 52.ŌĆŗ
    QTD (Quarter-to-Date):ŌĆŗ
    - Start date: First day of the current quarter (e.g., Q1TD starts on  Jan 1, Q2TD on Apr 1, Q3TD on Jul 1, and Q4TD on Oct 1).ŌĆŗ
    - End date: Latest <respective_date_column>. ŌĆŗ
    YTD (Year-to-Date):ŌĆŗ
    - Start date: January 1 of the current year.ŌĆŗ
    - End date: Latest <respective_date_column>.ŌĆŗ

    Date Calculation Guidelines
    - Do NOT use the current date for any calculations.
    whenever applicable.
  f. ALWAYS USE DATE_TRUNC() METHOD FOR MONTHLY TREND CALCULATIONS.
  g. Complete Time Period Rule: Always exclude incomplete time periods - for "past X months" use complete months only, ending at last day of previous complete month, not current incomplete month.
    Calculation Method: Use LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -1)) for end date and DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', MAX(date_column)), -(X+1))) for start date to ensure complete month boundaries.

  EXAMPLES:

  **Example 1: "Past 12 months" RX only**
  ```sql
  WHERE rx.fill_dt BETWEEN 
    DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
      (SELECT MAX(fill_dt) FROM rx_table)), -13))
  AND 
    LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
      (SELECT MAX(fill_dt) FROM rx_table)), -2))
  ```

  **Example 2: "Past 6 months" combined RX + MX**
  Use UNION ALL with separate date calculations:
  ```sql
  -- RX part
  SELECT ... FROM rx_table
  WHERE fill_dt BETWEEN 
    DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
      (SELECT MAX(fill_dt) FROM rx_table)), -7))
    AND LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH', 
      (SELECT MAX(fill_dt) FROM rx_table)), -2))

  UNION ALL

  -- MX part  
  SELECT ... FROM mx_table
  WHERE svc_dt BETWEEN 
    DATE_TRUNC('MONTH', ADD_MONTHS(DATE_TRUNC('MONTH', 
      (SELECT MAX(svc_dt) FROM mx_table)), -7))
    AND LAST_DAY(ADD_MONTHS(DATE_TRUNC('MONTH',

  ```

11. MULTI-BATCH COMPARISON RULES

  DEFINITION:
    Compare claims data across multiple PT cycle IDs (batches) side by side without calculating restatement percentages. 
    Shows raw claim counts for each batch to analyze trends across batches.

  WHEN TO USE:
    - User asks to "compare across X batches" without mentioning "restatement" or "percentage"
    - User asks for "trend across batches" without "capture ratio" keyword
    - User mentions "latest X batches" or "last N batches" (where X or N > 2)
    - User asks "how many claims in each batch"
    - User asks "show claims for each batch"
    - Keywords: "compare batches", "across batches", "each batch", "batch-level", "trend across batches"

  DO NOT USE WHEN:
    - User asks for "restatement" or "percentage change" ŌåÆ Use Section 10 (Restatement)
    - User EXPLICITLY asks for "capture ratio" or "projection factor" ŌåÆ Use Section 9 (NSP)
    - User asks for date-based monthly/quarterly trends (not batch-based) ŌåÆ Use standard date queries

  CRITICAL DISTINCTION:
    - "Compare TRODELVY trend across latest 3 batches" ŌåÆ Section 11 (Multi-Batch Comparison)
    - "What is TRODELVY capture ratio trend" ŌåÆ Section 9 (NSP Capture Ratio)
    - "TRODELVY restatement between 2 batches" ŌåÆ Section 10 (Restatement Analysis)

  MANDATORY LOGIC:

    a. BATCH SELECTION:
      - Default: Latest 3 batches (user can specify different number)
      - Identify pt_cycle_ids using ROW_NUMBER() ordered DESC
      - Include all N batches in results
      - Query handles up to latest 3 batches from preprocessed tables

    b. QUERY STRUCTURE:
      - Use CTE to get batch list with ROW_NUMBER()
      - Filter WHERE pt_cycle_id IN (SELECT from batch CTE)
      - Group by pt_cycle_id (and drug if drug-specific)
      - Order by pt_cycle_id DESC (latest first)
      - DO NOT calculate percentages or deltas

    c. OUTPUT COLUMNS:
      - pt_cycle_id (batch ID)
      - claim_count (total claims for that batch)
      - drug_name/regimen (if drug-specific query)
      - Optional: month breakdown if user requests "monthly by batch"

    d. NO PERCENTAGE CALCULATIONS:
      - This is NOT restatement analysis
      - Only show raw claim counts per batch
      - Let user see side-by-side comparison visually

  QUERY TEMPLATES:

    A. RX MULTI-BATCH COMPARISON (GENERAL):
    ```sql
    WITH batch_ids AS (
      SELECT 
        pt_cycle_id,
        ROW_NUMBER() OVER (ORDER BY pt_cycle_id DESC) as batch_rank
      FROM `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`stg_iqvia_laad_rx_onc_hist_preprocessed`
      GROUP BY pt_cycle_id
    )
    SELECT 
      rx.pt_cycle_id,
      COUNT(rx.clam_id) as claim_count
    FROM `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`stg_iqvia_laad_rx_onc_hist_preprocessed` rx
    WHERE rx.pt_cycle_id IN (SELECT pt_cycle_id FROM batch_ids WHERE batch_rank <= 3)
      AND rx.clam_type = 'PD'
    GROUP BY rx.pt_cycle_id
    ORDER BY rx.pt_cycle_id DESC
    ```

    B. MX MULTI-BATCH COMPARISON (DRUG-SPECIFIC):
    ```sql
    WITH batch_ids AS (
      SELECT 
        pt_cycle_id,
        ROW_NUMBER() OVER (ORDER BY pt_cycle_id DESC) as batch_rank
      FROM `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`stg_iqvia_laad_mx_onc_hist_preprocessed`
      GROUP BY pt_cycle_id
    )
    SELECT 
      mx.pt_cycle_id,
      hcpcs.regimen,
      COUNT(mx.mx_clam_id) as claim_count
    FROM `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`stg_iqvia_laad_mx_onc_hist_preprocessed` mx
    LEFT JOIN `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`breast_cancer_drugs_hcpcs_codes_mapped` hcpcs
      ON mx.prcr_cd = hcpcs.procedure_code AND hcpcs.regimen = 'TRODELVY'
    LEFT JOIN `commercial-us-apo-onc-iia-dev`.`setuserv_adhocs`.`breast_cancer_drugs_ndc_codes_mapped` ndc
      ON mx.ndc11 = ndc.product_code AND ndc.regimen = 'TRODELVY'
    WHERE mx.pt_cycle_id IN (SELECT pt_cycle_id FROM batch_ids WHERE batch_rank <= 3)
      AND (hcpcs.regimen IS NOT NULL OR ndc.regimen IS NOT NULL)
    ```

    ```sql
    WITH batch_ids AS (
    ```
